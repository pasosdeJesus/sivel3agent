## Description

Implement a **progressive, incremental crawler and case detector** that can be deployed during the MVP and expanded post-MVP. Unlike the binary "all or nothing" approach, this delivers value immediately while providing a clear upgrade path.

**Why progressive:**
- MVP needs enough pre-alerts to demonstrate the marketplace
- Waiting for full post-MVP architecture delays time-to-market
- Incremental improvements keep the team motivated and focused
- Early feedback shapes the final architecture

**Key design principles:**
1. **Deploy early, improve often** ÔÇö start with 5-10 sources, add more weekly
2. **Results-first** ÔÇö prioritize sources with highest case density
3. **Data preservation** ÔÇö never delete `clean_text` (enables reÔÇĹclassification)
4. **Case detection integrated with extraction** ÔÇö no separate classifier step
5. **MVP first, post-MVP later** ÔÇö simple architecture now, modular later

### Phase 2: Integrated Case Detection (Now ÔÇö MVP Enhancement) Ôťů Deployed

**Status (June 19, 2026):** Case detector hardened and integrated into the pipeline.

**Pipeline:** `scrape-news.ts Ôćĺ detect-cases.ts Ôćĺ generate-and-send.ts`

1. **`scrape-news.ts`** populates `source` with `clean_text`, `is_relevant`, `classification_reason`
2. **`detect-cases.ts`** reads `source WHERE detected_at IS NULL`, calls `detectCase()`, stores valid cases in `pre_alert` (status `pending`)
3. **`generate-and-send.ts`** reads `pre_alert WHERE status='pending'`, signs with agent wallet, sends to `sivel.xyz/api/pre-alerts/sync`, marks `status='synced'`

## Data Model

### `source` table (updated for progressive approach)

```sql
CREATE TABLE source (
    id SERIAL PRIMARY KEY,
    url VARCHAR(500) UNIQUE NOT NULL,
    content_hash VARCHAR(66) UNIQUE,
    clean_text TEXT,
    is_case BOOLEAN DEFAULT NULL,           -- NULL = pending, TRUE = case, FALSE = context
    case_reason TEXT,                       -- why it was classified
    category VARCHAR(50),                   -- Noche y Niebla code (if case)
    victim_count INTEGER,                   -- number of victims (if case)
    medium VARCHAR(100),
    title VARCHAR(500),
    published_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_source_is_case ON source(is_case);
CREATE INDEX idx_source_url ON source(url);
CREATE INDEX idx_source_content_hash ON source(content_hash);
```

## Crawler Technology Evaluation (June 2026)

### RSS-Only: Insufficient for Full Text

Initial RSS scraper with 10 sources collected 191 articles, but quality analysis revealed a critical gap:

| Source | Articles | Avg chars | Full text? |
|--------|----------|-----------|------------|
| Verdad Abierta | 15 | 9,803 | Ôťů `content:encoded` |
| HRW | 20 | 4,442 | Ôťů `content:encoded` |
| ReliefWeb | 20 | 3,863 | Ôťů `content:encoded` |
| INDEPAZ | 15 | 300 | ÔŁî Excerpt only |
| Comisi├│n Intereclesial | 9 | 347 | ÔŁî Excerpt only |
| Las2Orillas | 92 | 357 | ÔŁî Excerpt only |
| MiPutumayo | 10 | 367 | ÔŁî Excerpt only |
| El Tiempo | 10 | 128 | ÔŁî Excerpt only |

Only 3/10 sources provide full text via RSS. The rest return short excerpts (100-400 chars) ÔÇö insufficient for LLM extraction.

### Puppeteer (Standalone): Too Fragile

Attempted `puppeteer` directly for JavaScript-heavy sites:
- Ôťů Works for INDEPAZ and Comisi├│n Intereclesial monthly archives
- ÔŁî Connection drops, timeouts, 0 articles scraped in several runs
- ÔŁî No resume capability ÔÇö restart loses all progress
- ÔŁî Manual concurrency management needed

### Crawlee: Right Tool, Wrong Timing

Evaluated `crawlee` (CheerioCrawler + PlaywrightCrawler):
- Ôťů CheerioCrawler for static WordPress sites ÔÇö 10├Ś faster than Puppeteer
- Ôťů RequestQueue persists to disk, survives crashes
- Ôťů Auto-scaling concurrency
- Ôťů Combines static (Cheerio) + dynamic (Playwright) in same script
- ÔŁî Crawler followed share-button links recursively, diluting results
- ÔŁî Router/label mechanism needs debugging for article detail pages
- **Verdict:** Right tool for Phase 1 historical crawler, but needs dedicated debugging session

### Solution: Full-Text RSS via Article Fetch

Instead of trusting RSS content, the scraper now:
1. Parses RSS for article URLs + metadata (title, date, medium)
2. Fetches the original article URL with `axios` + `cheerio`
3. Extracts full text from `div.entry-content` or equivalent
4. Stores complete article text in `source.clean_text`

This converts the 100+ excerpt-only articles into full-text sources without browser overhead.
