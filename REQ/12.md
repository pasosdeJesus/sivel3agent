## Description

Implement a unified, modular architecture for `sivel3agent` that separates concerns and supports scalable ingestion. The immediate goal is to detect **new cases not already in the Banco de Datos** by crawling historical sources from **2026‑01‑01 onward**.

**Why start at 2026‑01‑01:**
- Banco de Datos already covers cases through 2025 — we must not duplicate them
- Focuses on fresh violence data for pre‑alert generation
- Progressive plan: 2026 first, expand backward to 2025‑07 later (#13)

**Source domains:** `doc/posiblesfuentes.txt` lists 168 domains historically used by Banco de Datos documenters. We prioritize WordPress sites for ease of crawling (monthly archives).

**This is a Post-MVP task.** Current MVP scraper (RSS, ~10 articles/feed) suffices for hackathon.

---

## Core Principles

1. **Separate concerns:** Crawler → Detector → Sync are independent modules
2. **Avoid duplication:** Crawl only 2026‑01‑01+ to not replicate existing Banco de Datos cases
3. **Case detection:** A "case" requires (a) victims AND (b) Noche y Niebla category
4. **Preserve data:** `clean_text` is never deleted — re‑classification is always possible
5. **Context vs case:** Articles without victims+category are context (stored as `is_case=false`)
6. **Results‑first order:** Crawler first (Phase 1), then detector (Phase 2)

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│                    Crawler Layer                         │
│  ┌──────────────────┐  ┌─────────────────────────────┐  │
│  │ RSS Scraper       │  │ Historical Crawler           │  │
│  │ (recent, ongoing) │  │ (2026-01-01+, WordPress     │  │
│  │                    │  │  monthly archives, resume)  │  │
│  └────────┬─────────┘  └──────────────┬──────────────┘  │
│           └──────────┬─────────────────┘                 │
│                      ▼                                   │
│              source table (raw articles)                 │
│              is_case = NULL (pending)                    │
└─────────────────────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│                 Case Detector Layer                      │
│  detectCase(article):                                    │
│    1. Fast keyword filter (violence + region)            │
│    2. LLM extraction (extractPreAlert)                   │
│    3. Validate: has victims? has category?               │
│    → YES: is_case=true, insert pre_alert                 │
│    → NO:  is_case=false, case_reason set                 │
│  clean_text preserved for all articles                   │
└─────────────────────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│                  Sync Layer                              │
│  pre_alert table → generate-and-send.ts → sivel.xyz     │
└─────────────────────────────────────────────────────────┘
```

## Implementation Order (Results‑First)

### Phase 1: Historical Crawler (immediate results)

**Goal:** Scrape 2026‑01‑01+ articles from WordPress sources used by Banco de Datos.

1. Select high‑density WordPress sources from `doc/posiblesfuentes.txt`:
   - `verdadabierta.com` ✅ already integrated
   - `justiciaypazcolombia.com` ✅ already integrated
   - `indepaz.org.co` ✅ already integrated
   - `conexionputumayo.com` ✅ RSS confirmed
   - `prensarural.org` ✅ SPIP confirmed
   - `elespectador.com` — WordPress
   - `elcolombiano.com` — WordPress
   - `lasillavacia.com` — WordPress
   - `contagioradio.com` — WordPress
   - `las2orillas.co` — WordPress
   - `vanguardia.com` — WordPress
   - `semana.com` — WordPress
   - `miputumayo.com.co` ✅ already integrated
2. Create `lib/crawler/wordpress.ts` — generic WordPress monthly archive crawler
3. Create `lib/crawler/spip.ts` — SPIP pagination crawler (Prensa Rural)
4. Create `config/sources-2026.json` with base URLs, selectors, rate limits
5. Create `scripts/crawl-2026.ts` — iterates months 2026‑01 through current, scrapes all articles, stores in `source` with `clean_text`, dedup by URL
6. Run → expect hundreds of articles from diverse regions (Cauca, Antioquia, Nariño, Arauca, Chocó, etc.)

**Expected output:** 500‑2000 articles from 10+ WordPress sources across Colombia.

### Phase 2: Case Detector (after crawler has data)

**Goal:** Process crawled articles, detect cases, generate pre‑alerts.

1. Create `lib/case-detector/index.ts` with `detectCase()`:
   - Fast keyword filter (reuse `classifyByKeywords`)
   - LLM extraction via `extractPreAlert` (#6)
   - Validate: `relatos[0].actos` has category AND (`personas` or `victimas` has entries)
   - Return `{ isCase: boolean, reason: string, category?: string, json?: object, victimCount?: number }`
2. Create `scripts/detect-cases.ts` — processes `is_case IS NULL` articles in batches
3. Valid cases → `is_case = true` → insert into `pre_alert` → ready for sync
4. Non‑cases → `is_case = false` → `case_reason` set → `clean_text` preserved

**Expected output:** 50‑200 pre‑alerts from 2026‑01‑01+ across Colombia.

### Phase 3: Data Model Migration

**Goal:** Add columns needed for case detection.

```sql
ALTER TABLE source ADD COLUMN is_case BOOLEAN DEFAULT NULL;
ALTER TABLE source ADD COLUMN case_reason TEXT;

-- Note: is_relevant from MVP #12 is superseded by is_case
-- classification_reason is superseded by case_reason
```

**Design notes:**
- `clean_text` is **never cleared** (enables re‑classification when models improve)
- Classification metadata (`category`, `victim_type`, `victim_count`) lives in `pre_alert.json_data`, not duplicated in `source`
- `source` stores: url, content_hash, clean_text, medium, title, published_at, metadata JSONB, is_case, case_reason

### Phase 4: Crawler Queue (scale/automation)

**Goal:** Job queue with resume for production crawling.

```sql
CREATE TABLE crawler_queue (
    id SERIAL PRIMARY KEY,
    source_name VARCHAR(100) NOT NULL,
    source_type VARCHAR(20) NOT NULL,    -- 'wordpress', 'spip', 'rss'
    base_url TEXT NOT NULL,
    config JSONB,                         -- selectors, rate limits
    status VARCHAR(20) DEFAULT 'pending', -- pending/running/done/error
    priority INTEGER DEFAULT 0,
    progress JSONB,                       -- {last_month, last_page, total_processed}
    last_run_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);
```

1. `scripts/run-crawler.ts` — time‑limited, resumable from `crawler_queue.progress`
2. Cron: every 2 hours, stops after 10 min processing

### Phase 5: Orchestration & Cleanup

- Cron pipeline: `crawl-2026.ts` → `detect-cases.ts` → `generate-and-send.ts`
- Remove old monolithic scripts
- Update `ARCHITECTURE.md` and `AGENTS.md`
