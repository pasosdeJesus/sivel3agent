## Description

Implement a **progressive, incremental crawler and case detector** that can be deployed during the MVP and expanded post-MVP. Unlike the binary "all or nothing" approach, this delivers value immediately while providing a clear upgrade path.

**Why progressive:**
- MVP needs enough pre-alerts to demonstrate the marketplace
- Waiting for full post-MVP architecture delays time-to-market
- Incremental improvements keep the team motivated and focused
- Early feedback shapes the final architecture

**Key design principles:**
1. **Deploy early, improve often** — start with 5-10 sources, add more weekly
2. **Results-first** — prioritize sources with highest case density
3. **Data preservation** — never delete `clean_text` (enables re‑classification)
4. **Case detection integrated with extraction** — no separate classifier step
5. **MVP first, post-MVP later** — simple architecture now, modular later

### Phase 2: Integrated Case Detection (Now — MVP Enhancement)

**Goal:** Replace separate classification + extraction with unified case detection.

**Current problem:** `extract-and-save.ts` processes all `is_relevant=true` articles, but many don't become valid cases.

**Solution:** `detectCase()` combines classification + extraction + validation.

**Implementation:**

```typescript
// lib/case-detector.ts
export async function detectCase(article: Article): Promise<{
  isCase: boolean;
  reason: string;
  json?: object;
  category?: string;
  victimCount?: number;
}> {
  // 1. Fast filter: violence + region (avoid LLM for obvious non-cases)
  const quick = classifyByKeywords(article.clean_text, article.title);
  if (!quick.relevant) {
    return { isCase: false, reason: quick.reason };
  }

  // 2. Attempt LLM extraction
  let json: object;
  try {
    json = await extractPreAlert(article);
  } catch {
    return { isCase: false, reason: 'LLM extraction failed' };
  }

  // 3. Validate: has violence category AND victims?
  const relato = json.relatos?.[0];
  if (!relato) {
    return { isCase: false, reason: 'No relato in JSON' };
  }

  const hasCategory = relato.actos?.some(a => a.agresion_particular !== null);
  const hasVictims = relato.personas?.length > 0 || relato.victimas?.length > 0;

  if (!hasCategory) {
    return { isCase: false, reason: 'No Noche y Niebla category (actos)' };
  }

  if (!hasVictims) {
    return { isCase: false, reason: 'No victims (individual or collective)' };
  }

  const category = relato.actos?.[0]?.agresion_particular;
  const victimCount = relato.personas?.length || relato.victimas?.length || 0;

  return {
    isCase: true,
    reason: 'Valid case detected',
    json,
    category,
    victimCount,
  };
}
```

## Data Model

### `source` table (updated for progressive approach)

```sql
CREATE TABLE source (
    id SERIAL PRIMARY KEY,
    url VARCHAR(500) UNIQUE NOT NULL,
    content_hash VARCHAR(66) UNIQUE,
    clean_text TEXT,
    is_case BOOLEAN DEFAULT NULL,           -- NULL = pending, TRUE = case, FALSE = context
    case_reason TEXT,                       -- why it was classified
    category VARCHAR(50),                   -- Noche y Niebla code (if case)
    victim_count INTEGER,                   -- number of victims (if case)
    medium VARCHAR(100),
    title VARCHAR(500),
    published_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_source_is_case ON source(is_case);
CREATE INDEX idx_source_url ON source(url);
CREATE INDEX idx_source_content_hash ON source(content_hash);
```

**Note:** `is_relevant` and `classification_reason` from MVP #12 are **deprecated** (replaced by `is_case` and `case_reason`). Existing columns remain for backward compatibility.

### `pre_alert` table (unchanged)

```sql
CREATE TABLE pre_alert (
    id SERIAL PRIMARY KEY,
    event_hash VARCHAR(66) UNIQUE NOT NULL,
    json_data JSONB NOT NULL,
    status VARCHAR(20) DEFAULT 'pending',
    source_id INTEGER REFERENCES source(id),
    created_at TIMESTAMP DEFAULT NOW()
);
```

## Crawler Technology Evaluation (June 2026)

### RSS-Only: Insufficient for Full Text

Initial RSS scraper with 10 sources collected 191 articles, but quality analysis revealed a critical gap:

| Source | Articles | Avg chars | Full text? |
|--------|----------|-----------|------------|
| Verdad Abierta | 15 | 9,803 | ✅ `content:encoded` |
| HRW | 20 | 4,442 | ✅ `content:encoded` |
| ReliefWeb | 20 | 3,863 | ✅ `content:encoded` |
| INDEPAZ | 15 | 300 | ❌ Excerpt only |
| Comisión Intereclesial | 9 | 347 | ❌ Excerpt only |
| Las2Orillas | 92 | 357 | ❌ Excerpt only |
| MiPutumayo | 10 | 367 | ❌ Excerpt only |
| El Tiempo | 10 | 128 | ❌ Excerpt only |

Only 3/10 sources provide full text via RSS. The rest return short excerpts (100-400 chars) — insufficient for LLM extraction.

### Puppeteer (Standalone): Too Fragile

Attempted `puppeteer` directly for JavaScript-heavy sites:
- ✅ Works for INDEPAZ and Comisión Intereclesial monthly archives
- ❌ Connection drops, timeouts, 0 articles scraped in several runs
- ❌ No resume capability — restart loses all progress
- ❌ Manual concurrency management needed

### Crawlee: Right Tool, Wrong Timing

Evaluated `crawlee` (CheerioCrawler + PlaywrightCrawler):
- ✅ CheerioCrawler for static WordPress sites — 10× faster than Puppeteer
- ✅ RequestQueue persists to disk, survives crashes
- ✅ Auto-scaling concurrency
- ✅ Combines static (Cheerio) + dynamic (Playwright) in same script
- ❌ Crawler followed share-button links recursively, diluting results
- ❌ Router/label mechanism needs debugging for article detail pages
- **Verdict:** Right tool for Phase 1 historical crawler, but needs dedicated debugging session

### Solution: Full-Text RSS via Article Fetch

Instead of trusting RSS content, the scraper now:
1. Parses RSS for article URLs + metadata (title, date, medium)
2. Fetches the original article URL with `axios` + `cheerio`
3. Extracts full text from `div.entry-content` or equivalent
4. Stores complete article text in `source.clean_text`

This converts the 100+ excerpt-only articles into full-text sources without browser overhead.

### Anti-Hallucination Detector (June 18, 2026)

Added post-extraction validation in `lib/case-detector/index.ts`:
- Extracts proper names from LLM output using regex
- Verifies each name appears in source text
- Flags known hallucination patterns ("Juan Pérez", "José Mendoza", "vereda X", "municipio de Y")
- Rejects pre-alerts with invented information

**Model comparison:**

| Model | Temp | Pre-alerts | Clean cases | Notes |
|-------|------|-----------|-------------|-------|
| qwen2.5:7b | 0.2 | 96 | 3 | High volume, many hallucinations |
| qwen2.5:7b | 0.0 | 8 | 3 | Lower volume, still hallucinates |
| **qwen3:8b** | **0.2** | **5** | **5** | **Zero hallucinations detected** ✅ |

**Winner for production: `qwen3:8b`** — 100% clean rate with anti‑hallucination filter. Slower (23s vs 7s) but trustworthy. 7B with temp 0 still hallucinates due to GGUF quantization non‑determinism.
