## Description

Implement a **progressive, incremental crawler and case detector** that can be deployed during the MVP and expanded post-MVP. Unlike the binary "all or nothing" approach, this delivers value immediately while providing a clear upgrade path.

**Why progressive:**
- MVP needs enough pre-alerts to demonstrate the marketplace
- Waiting for full post-MVP architecture delays time-to-market
- Incremental improvements keep the team motivated and focused
- Early feedback shapes the final architecture

**Key design principles:**
1. **Deploy early, improve often** â€” start with 5-10 sources, add more weekly
2. **Results-first** â€” prioritize sources with highest case density
3. **Data preservation** â€” never delete `clean_text` (enables reâ€‘classification)
4. **Case detection integrated with extraction** â€” no separate classifier step
5. **MVP first, post-MVP later** â€” simple architecture now, modular later

## Data Model

### `source` table (updated for progressive approach)

```sql
CREATE TABLE source (
    id SERIAL PRIMARY KEY,
    url VARCHAR(500) UNIQUE NOT NULL,
    content_hash VARCHAR(66) UNIQUE,
    clean_text TEXT,
    is_case BOOLEAN DEFAULT NULL,           -- NULL = pending, TRUE = case, FALSE = context
    case_reason TEXT,                       -- why it was classified
    category VARCHAR(50),                   -- Noche y Niebla code (if case)
    victim_count INTEGER,                   -- number of victims (if case)
    medium VARCHAR(100),
    title VARCHAR(500),
    published_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_source_is_case ON source(is_case);
CREATE INDEX idx_source_url ON source(url);
CREATE INDEX idx_source_content_hash ON source(content_hash);
```

**Note:** `is_relevant` and `classification_reason` from MVP #12 are **deprecated** (replaced by `is_case` and `case_reason`). Existing columns remain for backward compatibility.

## Crawler Technology Evaluation (June 2026)

### RSS-Only: Insufficient for Full Text

Initial RSS scraper with 10 sources collected 191 articles, but quality analysis revealed a critical gap:

| Las2Orillas | 92 | 357 | âťŚ Excerpt only |
| MiPutumayo | 10 | 367 | âťŚ Excerpt only |
| El Tiempo | 10 | 128 | âťŚ Excerpt only |

Only 3/10 sources provide full text via RSS. The rest return short excerpts (100-400 chars) â€” insufficient for LLM extraction.

### Puppeteer (Standalone): Too Fragile

Attempted `puppeteer` directly for JavaScript-heavy sites:
- âś… Works for INDEPAZ and ComisiĂłn Intereclesial monthly archives
- âťŚ Connection drops, timeouts, 0 articles scraped in several runs
- âťŚ No resume capability â€” restart loses all progress
- âťŚ Manual concurrency management needed

### Crawlee: Right Tool, Wrong Timing

Evaluated `crawlee` (CheerioCrawler + PlaywrightCrawler):
- âś… CheerioCrawler for static WordPress sites â€” 10Ă— faster than Puppeteer
- âś… RequestQueue persists to disk, survives crashes
- âś… Auto-scaling concurrency
- âś… Combines static (Cheerio) + dynamic (Playwright) in same script
- âťŚ Crawler followed share-button links recursively, diluting results
- âťŚ Router/label mechanism needs debugging for article detail pages
- **Verdict:** Right tool for Phase 1 historical crawler, but needs dedicated debugging session

### Solution: Full-Text RSS via Article Fetch

Instead of trusting RSS content, the scraper now:
1. Parses RSS for article URLs + metadata (title, date, medium)
2. Fetches the original article URL with `axios` + `cheerio`
3. Extracts full text from `div.entry-content` or equivalent
4. Stores complete article text in `source.clean_text`

This converts the 100+ excerpt-only articles into full-text sources without browser overhead.
