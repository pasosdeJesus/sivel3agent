## Dataset

### Source

The dataset is built from articles that have already been classified as **relevant** (`is_relevant = true`) in the `source` table (from #12 and #13). These articles come from high‑value sources:

| Source | Type | Region |
|--------|------|--------|
| INDEPAZ | RSS + historical | Colombia |
| Verdad Abierta | Historical | Colombia |
| Comisión Intereclesial | RSS + historical | Colombia |
| Prensa Rural | Historical | Colombia |
| MiPutumayo Noticias | RSS + historical | Putumayo |
| El Espectador | Web scraping (with filter) | Colombia |

**Size:** 500–1000 examples per model. We will select a balanced set covering:

- **Regions:** Putumayo, Cauca, Antioquia, Nariño, Bogotá (from #10)
- **Violation types:** All Noche y Niebla categories (DD.HH., VPS, DIHC, Acción Bélica)
- **Lengths:** Short (300–1000 words), medium (1000–3000 words), long (>3000 words)
- **Sources:** Mix from #9 and #13 (RSS + historical)

### Annotation Process

For each selected article, we will manually create the expected JSON following `relato.schema.json`. The schema includes fields defined by the Noche y Niebla conceptual framework:

- `titulo`: Short summary
- `hechos`: Description (who + what + to whom + circumstances)
- `fecha`: Date of event
- `departamento`, `municipio`: Geographic location
- `personas`: Victims or affected individuals
- `grupos`: Responsible groups (armed groups, state forces, etc.)
- `actos`: Type of aggression (e.g., extrajudicial execution, forced displacement)
- `contextos`: Contextual factors (e.g., persecution of social organizations, peace processes)

**Annotation tool:** We will use a simple JSONL format where each line contains:

```json
{
  "input": "Título: {title}\nFecha: {date}\nFuente: {source}\nTexto: {clean_text}",
  "output": "{... JSON conforming to relato.schema.json ...}"
}
```

**Quality control:** 20% of the dataset will be double‑annotated by a second human to ensure consistency and catch errors.

## Fine‑tuning Strategy

### Technique: LoRA (Low‑Rank Adaptation)

We will use **LoRA** (or QLoRA) for efficient fine‑tuning. This reduces VRAM requirements and training time while achieving results close to full fine‑tuning.

| Parameter | Value | Reason |
|-----------|-------|--------|
| **Rank (r)** | 16 | Balanced between adaptability and efficiency |
| **Alpha** | 32 | Scaling factor for LoRA weights |
| **Dropout** | 0.1 | Prevents overfitting |
| **Learning rate** | 2e‑4 | Standard for LoRA fine‑tuning |
| **Batch size** | 4 | Fits in 16 GB VRAM |
| **Epochs** | 3 | Enough for convergence without overfitting |
| **Optimizer** | AdamW | Standard |
| **Quantization** | 4‑bit (QLoRA) | Reduces VRAM usage, enables fine‑tuning on RX 9060 XT |

### Training Environment

| Option | Pros | Cons | Recommendation |
|--------|------|------|----------------|
| **Local (RX 9060 XT)** | Full control, no data transfer | Requires ROCm setup, may be slower | ✅ **Primary** – model is already local |
| **TuneKit (Colab)** | Free, fast (15 min), no setup | Requires dataset upload, limited GPU time | ✅ **Fallback** – if local fails |
| **Hugging Face AutoTrain** | Simple, integrated | May not support LoRA, costs | ❌ Not recommended |

**We will start locally.** If VRAM is insufficient or training is too slow, we will fallback to TuneKit.

### Tools

- **Unsloth** (recommended): 2x faster, 70% less VRAM than standard LoRA.
  - Installation: `pip install unsloth`
  - Works with QLoRA and 4‑bit quantization.
- **Hugging Face PEFT** (fallback): More flexible but slower.

## Implementation Steps

### 1. Prepare Dataset

Create `scripts/prepare-finetune-dataset.ts`:

```typescript
// scripts/prepare-finetune-dataset.ts
import { db } from '../lib/db';
import fs from 'fs';

async function prepareDataset() {
  const sources = await db
    .selectFrom('source')
    .select(['title', 'published_at', 'clean_text', 'medium', 'url'])
    .where('is_relevant', '=', true)
    .where('clean_text', 'is not', null)
    .orderBy('published_at', 'desc')
    .limit(1500)
    .execute();

  const jsonl = sources.map(s => {
    const input = `Título: ${s.title}\nFecha: ${s.published_at?.toISOString().split('T')[0] || ''}\nFuente: ${s.medium} (${s.url})\nTexto: ${s.clean_text}`;
    // output JSON must be manually annotated – this script only generates the template
    return { input, output: '{}' };
  });

  fs.writeFileSync('data/finetune-dataset-template.jsonl', jsonl.map(JSON.stringify).join('\n'));
  console.log(`Dataset template created with ${jsonl.length} entries.`);
}

prepareDataset().catch(console.error);
```

**Annotation workflow:**
1. Run `prepare-finetune-dataset.ts` to generate the template.
2. Annotate each `output` field manually (or using a tool like Doccano).
3. Save as `data/finetune-dataset.jsonl`.

### 2. Fine‑tuning Script (Unsloth)

Create `scripts/finetune.py` (Python script, run with `uv` or `python`):

```python
# scripts/finetune.py
import json
from unsloth import FastLanguageModel
import torch
from trl import SFTTrainer
from transformers import TrainingArguments

# Configuration
MODEL_NAME = "Qwen/Qwen2.5-7B-Instruct"  # or "Qwen/Qwen3-8B-Instruct"
OUTPUT_DIR = "lora-adapter-7b"  # or "lora-adapter-8b"

# Load model
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name=MODEL_NAME,
    max_seq_length=2048,
    dtype=None,  # auto-detect
    load_in_4bit=True,
)

# Add LoRA adapters
model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"],
    lora_alpha=16,
    lora_dropout=0,
    bias="none",
    use_gradient_checkpointing="unsloth",
    random_state=3407,
)

# Load dataset
dataset = []
with open("data/finetune-dataset.jsonl", "r") as f:
    for line in f:
        dataset.append(json.loads(line))

# Format as chat messages
def format_example(ex):
    return [
        {"role": "user", "content": ex["input"]},
        {"role": "assistant", "content": ex["output"]}
    ]

# Train
trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    train_dataset=dataset,
    formatting_func=format_example,
    args=TrainingArguments(
        per_device_train_batch_size=4,
        gradient_accumulation_steps=4,
        warmup_steps=5,
        max_steps=60,
        learning_rate=2e-4,
        fp16=not torch.cuda.is_bf16_supported(),
        bf16=torch.cuda.is_bf16_supported(),
        logging_steps=1,
        optim="adamw_8bit",
        weight_decay=0.01,
        lr_scheduler_type="linear",
        seed=3407,
        output_dir="outputs",
    ),
)

trainer.train()

# Save LoRA adapters
model.save_pretrained(OUTPUT_DIR)
print(f"✅ LoRA adapters saved to {OUTPUT_DIR}")
```
