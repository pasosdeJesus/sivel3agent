# REQ/11 — Fine‑Tuning for Noche y Niebla Classification

Fine‑tune the best candidate model to improve structured JSON extraction from news articles, with focus on the two weakest metrics from #10 evaluation:
1. **Aggression code classification** (currently 71% for both top models)
2. **Group identification** (currently 0‑14%)

**Decision:** Postponed for MVP. See rationale below. Production testing confirmed this is correct — `qwen3:8b` at 0% hallucination is adequate. Expanding golden dataset is higher priority than fine‑tuning.

Full evaluation results in `doc/article-choosing-llm.md` and REQ/10.

## Why Postponed

1. **No 14B advantage.** The larger model performs worse (43% vs 71%). Fine‑tuning a 14B that underperforms 7B is not worth the investment.

2. **8B production validated.** Zero hallucinations, 36 pre‑alerts sent to sivel.xyz, 12+ departments covered. The current model is adequate for MVP.

3. **Dataset bottleneck.** Fine‑tuning requires 100‑200 annotated examples. We have 7 golden dataset cases + ~11 verified pre‑alerts from production. The pipeline itself (`detect-cases.ts`) can help build this dataset by flagging cases for human review.

4. **Better ROI alternatives exist.** Before investing in annotation + training:
   - Expand RSS feeds and crawler sources (#12 Phase 4)
   - Better prompt engineering — SKILL‑based prompt already produces valid JSON 100% of the time
   - vLLM with guided JSON (#14) would force schema‑compliant output

5. **ROCm compatibility risk.** Unsloth has limited AMD support. Most LoRA/QLoRA tutorials assume CUDA. Google Colab (T4, free tier) is the fallback.

## Dataset Strategy

When we have enough annotated cases (target: 100‑200), the process is:

### Source

Cases from the Banco de Datos with verifiable press sources, same methodology as the 7 Putumayo cases:

1. Obtain XML export from CINEP/PPP (like `casos-put-2025-I.xrlat`)
2. Convert to JSON via `scripts/convert-xrlat.ts` (#15)
3. Match each case to its press source URL
4. Scrape the full article text via `scripts/scrape-golden.ts`
5. The Banco de Datos annotation becomes the ground truth `output`

### Semi‑automated annotation

1. Run the best current model (qwen3:8b) on each article to generate a draft JSON
2. Human reviewer corrects the draft (faster than annotating from scratch)
3. 20% double‑annotated for quality control

### Target diversity

- **Regions:** Putumayo, Cauca, Antioquia, Nariño, Bogotá, Palestina
- **Types:** DD.HH. (A), VPS (B), DIHC (D), Acción Bélica (C)
- **Periods:** 2020‑2026, focus on 2025‑2026
- **Sources:** Mix of national press, local media, NGO reports

## Fine‑Tuning Plan (When Ready)

### Candidate Model

**qwen2.5:7b-instruct** — best speed/precision balance (7.0s, 71%). qwen3:8b is 3× slower for same precision.

### Technique: QLoRA with Unsloth

| Parameter | Value |
|-----------|-------|
| Rank (r) | 16 |
| Alpha | 32 |
| Dropout | 0.1 |
| Learning rate | 2e‑4 |
| Batch size | 4 |
| Epochs | 3 |
| Quantization | 4‑bit (QLoRA) |
| Max seq length | 2048 |

### Environment

| Option | Feasibility |
|--------|-------------|
| Local RX 9060 XT | ⚠️ Unsloth ROCm support uncertain |
| Google Colab (T4) | ✅ Free, 15 GB VRAM, CUDA |
| TuneKit | ✅ Generates Colab notebook |

**Recommended:** Google Colab with T4 GPU (free tier). Upload dataset, run Unsloth, download LoRA adapter.

### Expected improvement

| Metric | Current | Target |
|--------|---------|--------|
| Aggression accuracy | 71% | >80% |
| Group identification | 0% | >40% |
| JSON validity | 100% | 100% |
| Speed | 7.0s | ~7.0s (LoRA adds negligible overhead) |

**Realistic target:** +10‑15% on aggression, +30‑40% on groups. The 71%→85% jump claimed in earlier versions is optimistic for a 7B model.
